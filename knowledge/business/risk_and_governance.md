# Risk & Governance Blueprint

This blueprint outlines the risk register, quality controls, and AI safety practices implemented to secure the integrity, data privacy, and trust of the **Career Intelligence Studio (CIS)** platform.

---

## 1. Governance Model

Governance is structured around three key principles:
1.  **Strict Data Isolation:** All user personal data, documents, and credentials must remain in localized storage unless explicitly consented to by the user.
2.  **Evidence Traceability (Fact-Grounded Generation):** Every claim, bullet point, or resume line generated by the system must map directly to historical source documents in the Career Workspace.
3.  **Human-in-the-Loop Override:** The AI never automatically commits changes or exports files without manual human review and approval.

---

## 2. Risk Register

Risks are categorized and prioritized based on **Likelihood (L)** and **Severity (S)** (Scale: Low, Medium, High).

| Risk ID | Category | Description | L | S | Governance Control |
|---|---|---|---|---|---|
| **RSK-AI-01** | AI Safety | The LLM fabricates employment history, skills, or achievements (hallucination). | High | High | **Grounding Engine Audits:** Parse text generation outputs, compare against the source document embeddings, and block elements that fail a 90% confidence threshold. |
| **RSK-PRV-02** | Privacy | Local user data is leaked through debug logs or third-party cloud LLM API data-retention policies. | Medium | High | **Zero-Retention API Contracts & Local Processing:** Enforce local embedding engines where possible. Ensure cloud API requests use strict zero-retention/opt-out configurations. |
| **RSK-FID-03** | Quality | The system exports DOCX resumes with broken margins, misaligned tables, or incorrect font weights. | High | Medium | **Fidelity Regression Suite:** Render exported documents to virtual PDF layout trees and verify style XML nodes remain unchanged. |
| **RSK-SEC-04** | Security | Unencrypted local SQLite database is accessed by other client applications on desktop or mobile. | Medium | High | **Data Volume Encryption:** Utilize OS-level sandbox directories and database-level encryption (e.g., SQLCipher) to prevent external file access. |
| **RSK-OPS-05** | Operations | Workspace schemas break during PostgreSQL migration, leading to synchronization failure. | Low | Medium | **Dual-Write Sandbox Testing:** Validate all schema migrations against a local PostgreSQL test instance prior to deploying changes. |

## 3. AI Safety Controls

### Evidence-Grounding Engine (Fact Checks)
*   **Vector Distance Check:** Suggested modifications must be validated using semantic search. If a sentence has a cosine distance greater than `0.3` (using the target embedding model) from the nearest source text block, it must be flagged for manual evidence reference.
*   **Automatic Claim Verification:** Prompt-level constraints must explicitly direct the model to cite matching source IDs:
    ```
    Every claim must end with a citation to the source document ID. 
    Format: [Claim] (Source: DOC-XXX).
    Do NOT invent dates, names, or metrics.
    ```

### Hallucination UI Safeguards
*   **Visual Confidence Flags:** Highlight any generated sentence in **yellow** if it has a low similarity score, and in **red** if no source citation can be resolved.
*   **Mandatory Acknowledgment:** The application blocks exports if any "red-flagged" (uncited) content exists, forcing the user to either write in the source data or explicitly acknowledge the exception.

---

## 4. Quality & Compliance Controls

### Document Fidelity Protection
*   **Style Extraction Hook:** Prior to editing, the engine extracts the template styling structure (`document.xml.rels`, `styles.xml`, and header/footer relationship configurations) and saves it as a lockfile.
*   **Structural Difference Audit:** After editing, the system compares the output file XML structural tree against the original lockfile. Any deviation in spacing, margin styles, or fonts triggers a compile error, preventing export.

---
